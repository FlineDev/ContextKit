# Subagent: check-localization

## Purpose
Perform comprehensive localization audit with execution flow pattern to ensure iOS/macOS applications meet international and accessibility standards from day one.

## Context Requirements
- Project files (SwiftUI views, UIKit controllers, Core Data models)
- Current feature specifications and technical plans
- Target platforms (iOS/macOS/tvOS/watchOS)
- String Catalog files (.xcstrings)
- Localization configuration files

## Execution Flow (subagent)

### Phase 1: Project Build and String Catalog Update
1. Execute project build to ensure compilation
   ‚Üí If build fails: ERROR "Fix compilation errors before localization audit"
   ‚Üí Build command: `swift build` (Package) or `xcodebuild` (App)
2. Update String Catalog with latest strings
   ‚Üí Force Xcode to regenerate .xcstrings files
   ‚Üí Scan for new LocalizedStringKey usage
   ‚Üí If String Catalog missing: WARN "Create String Catalog for localization"
3. Analyze generated string catalog content
   ‚Üí Extract all LocalizedStringKey entries
   ‚Üí Identify auto-generated vs custom string keys
   ‚Üí Map strings to source code locations

### Phase 2: String Catalog Analysis and Enhancement
4. Review contextual comments in String Catalog
   ‚Üí Compare Xcode auto-generated comments with usage context
   ‚Üí Identify insufficient or misleading comments
   ‚Üí Generate improved contextual descriptions for translators
5. Validate semantic localization keys
   ‚Üí Check for descriptive keys vs raw text keys
   ‚Üí Ensure keys follow semantic naming conventions
   ‚Üí Flag hardcoded strings that should use LocalizedStringKey
6. Analyze pluralization rules implementation
   ‚Üí Verify .stringsdict files or equivalent for plural forms
   ‚Üí Check language-specific plural rules (English, German, Russian, etc.)
   ‚Üí Validate count-based string variations

### Phase 3: Model Data and UI Localization Validation
7. Scan Core Data models for localization needs
   ‚Üí Identify user-facing string attributes
   ‚Üí Check for proper localization of model display names
   ‚Üí Validate entity and attribute descriptions for localization
8. Review SwiftUI/UIKit views for localization patterns
   ‚Üí Verify all user-facing text uses LocalizedStringKey
   ‚Üí Check accessibility labels and hints for localization
   ‚Üí Validate dynamic content localization (dates, numbers, currencies)
9. Audit region-specific formatters
   ‚Üí Date and time formatting with locale awareness
   ‚Üí Number formatting with proper separators and decimal points
   ‚Üí Currency formatting with appropriate symbols and positions
   ‚Üí Measurement unit conversions and display

### Phase 4: Constitutional Localizability Compliance
10. Apply constitutional principle: "Localizability from day one"
    ‚Üí Every user-facing string must be localizable
    ‚Üí No hardcoded text in user interface elements
    ‚Üí Proper semantic key naming for translator context
11. Validate platform-appropriate localization patterns
    ‚Üí iOS: String Catalog integration with Xcode
    ‚Üí macOS: NSLocalizedString with proper bundle handling
    ‚Üí Cross-platform: SwiftUI LocalizedStringKey consistency
12. Check accessibility localization compliance
    ‚Üí VoiceOver labels and hints properly localized
    ‚Üí accessibility identifiers language-neutral
    ‚Üí Dynamic type support across languages

### Phase 5: Advanced Localization Patterns
13. Analyze RTL (Right-to-Left) language support
    ‚Üí UI layout adaptability for Arabic, Hebrew
    ‚Üí Text alignment and reading direction
    ‚Üí Icon and image mirroring requirements
14. Review locale-specific UI adaptations
    ‚Üí Text expansion considerations (German, Finnish)
    ‚Üí Font and typography adjustments
    ‚Üí Cultural color and symbol appropriateness
15. Validate localization testing infrastructure
    ‚Üí Pseudolocalization setup for layout testing
    ‚Üí Locale switching capabilities in development
    ‚Üí Translation memory integration possibilities

## Constitutional Principles Integration

**Localizability from Day One**: Every aspect of the audit enforces the constitutional principle that applications must be designed for international audiences from initial development, not as an afterthought.

**Accessibility-First Design**: Localization audit includes comprehensive review of VoiceOver labels, accessibility hints, and dynamic type support across languages.

**Platform-Appropriate UX**: Ensures localization follows iOS Human Interface Guidelines and macOS design principles for international user experiences.

**Code Maintainability**: Promotes semantic localization key naming and proper architectural patterns that make ongoing localization maintenance sustainable.

- **"Build failed"** ‚Üí Project must compile before localization analysis

- **"Formatter locale issues"** ‚Üí Fix regional formatting configuration
- **"Constitutional violation"** ‚Üí Address hardcoded strings blocking localizability

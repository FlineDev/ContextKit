## Description
Handle Swift Package release workflow with GitHub integration, cross-platform validation, and constitutional compliance

## Parameters
- `release_type` (optional): major, minor, patch - defaults to detecting from changes
- `prerelease` (optional): alpha, beta, rc - for pre-release versions

## Execution Flow (main)
1. Validate Swift package release prerequisites
   ‚Üí Verify this is a Swift package project (Package.swift present)
   ‚Üí Check that all features are complete (no pending tasks in Context/Features/)
   ‚Üí Ensure git working directory is clean with all changes committed
   ‚Üí If not package project: ERROR "Use /Implement/release-app for iOS/macOS apps"
2. Run Swift package constitutional validation
   ‚Üí Execute API design review: Clear naming, protocol-oriented design, minimal dependencies
   ‚Üí Execute cross-platform compatibility: Linux, Windows (where applicable), macOS compatibility
   ‚Üí Execute documentation completeness: DocC documentation, code examples, API coverage
   ‚Üí Execute maintainability assessment: Clear architecture, comprehensive error handling
   ‚Üí If any violations: ERROR with specific remediation guidance
3. Determine release version and scope
   ‚Üí Analyze committed changes since last release (breaking changes ‚Üí major, features ‚Üí minor, fixes ‚Üí patch)
   ‚Üí Parse git history and conventional commits if used
   ‚Üí If release_type provided: validate against change scope and breaking changes
   ‚Üí If prerelease specified: append prerelease identifier to version
4. Update package metadata and documentation
   ‚Üí Update version in Package.swift if manually specified
   ‚Üí Update README.md with latest features and breaking changes
   ‚Üí Regenerate API documentation with DocC
   ‚Üí Update CHANGELOG.md with comprehensive release notes
5. Run comprehensive package validation
   ‚Üí Swift package resolution and build across all platforms
   ‚Üí Linux compatibility testing (if cross-platform package)
   ‚Üí Execute complete test suite with coverage analysis
   ‚Üí Validate public API surface and breaking change detection
   ‚Üí Performance regression testing for performance-critical packages
6. Documentation and example validation
   ‚Üí Ensure all public APIs have comprehensive documentation
   ‚Üí Validate code examples in documentation compile and run
   ‚Üí Check README.md installation and usage instructions are current
   ‚Üí Verify DocC documentation builds without warnings
7. Create release artifacts
   ‚Üí Generate GitHub release notes from CHANGELOG.md and git history
   ‚Üí Create comprehensive Context/Releases/[Version].md documentation
   ‚Üí Package any additional distribution artifacts (if needed)
   ‚Üí Prepare announcement content for developer community
8. GitHub release and tagging
   ‚Üí Create and push annotated git tag with release version
   ‚Üí Create GitHub release with generated release notes
   ‚Üí Upload any additional release artifacts to GitHub release
   ‚Üí Update GitHub repository description and topics if needed
9. Package registry and distribution
   ‚Üí Swift Package Index will automatically detect new releases
   ‚Üí Update any package manager registries if applicable
   ‚Üí Notify dependent projects of updates if breaking changes
   ‚Üí Update package discovery and showcase materials
10. Finalize release process
    ‚Üí Merge release branch to main if using git flow
    ‚Üí Update development branch with post-release version
    ‚Üí Create announcement for developer community
    ‚Üí Update package ecosystem integrations
11. Return: SUCCESS (Swift package released with full validation and documentation)

### Breaking Change Detection
- **Public API Changes**: Removed or modified public methods, properties, types
- **Behavioral Changes**: Different behavior for same input (even if compatible)
- **Dependency Changes**: Major version bumps in public dependencies
- **Platform Support**: Dropping support for previously supported platforms

## Package Validation Pipeline

### Build and Test Validation
```bash
# Multi-platform build validation
swift build -c release
swift test --parallel

# Linux compatibility (if cross-platform)
docker run --rm -v "$PWD:/workspace" -w /workspace swift:5.9 swift test

# Documentation building
swift package generate-documentation
```

### API Surface Analysis
```bash
# Generate and compare API surface
swift api-diff --baseline=${PREVIOUS_VERSION} --new=${NEW_VERSION}

# Validate public API documentation coverage  
swift package describe-api --format json | jq '.coverage'
```

## Error Conditions
- "Not a Swift package" ‚Üí Use /Implement/release-app for iOS/macOS apps
- "Constitutional violations" ‚Üí Fix API design, documentation, or compatibility issues
- "Breaking changes undocumented" ‚Üí Must provide clear migration guidance
- "Cross-platform failures" ‚Üí Fix Linux/Windows compatibility issues
- "Documentation incomplete" ‚Üí All public APIs must have comprehensive documentation
- "Test failures" ‚Üí All tests must pass before release
- "GitHub release errors" ‚Üí Fix repository access, tagging, or release creation issues
